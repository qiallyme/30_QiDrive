> ## Why this is the right shape
> 
> A node like `Energy Rhythm` stays a **node**.  
> Its meaning stays in **blocks**:
> 
> - `thesis` → “Energy is wave-based, not evenly sustained”
> - `day_to_day` → “Can be highly productive in bursts”

> ## AI connection layer
> 
> Supabase’s AI stack is built around Postgres + `pgvector`, and their docs explicitly support structured embeddings, automatic embedding workflows, permission-aware RAG with RLS, and integrations like LangChain. Their automatic embeddings guide uses async infrastructure rather than doing embedding work inline with writes, which is the right idea here. citeturn246368view0turn517138view3turn517138view0turn517138view1turn246368view1
> 
> So the flow should be:
> 
> 1. User edits `manual_nodes` / `manual_blocks`
> 2. A webhook or trigger queues chunk regeneration
> 3. Edge Function rebuilds `manual_chunks`
> 4. Edge Function generates embeddings
> 5. AI queries `match_manual_chunks(...)`
> 6. RLS keeps retrieval permission-safe
> 
> Supabase’s database webhooks fire on `INSERT`, `UPDATE`, and `DELETE`, and they’re basically async trigger wrappers over `pg_net`, which is useful when you want changes to kick off external work without blocking writes. citeturn775291view0

> - one giant markdown column
> - one giant JSON blob
> - a separate vector DB

>
