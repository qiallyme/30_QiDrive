> Conflict Resolution with CRDTs

> In decentralized, real-time co-editing environments, a significant challenge arises when multiple users modify the same data concurrently, potentially leading to conflicts.31 Anytype addresses this by leveraging “Conflict-free Replicated Data Types (CRDTs)” to ensure data consistency across devices without requiring complex conflict resolution mechanisms or user intervention.32 CRDTs are specifically designed so that different replicas can execute concurrent operations in any order and still converge to the same consistent state when the system becomes quiescent.32

> 

> Any-sync, Anytype’s synchronization protocol, is designed to synchronize digital objects structured as Conflict-free Replicated Data Directed Acyclic Graphs (DAGs).33 When situations arise where objects have multiple “heads” (representing concurrent changes from different devices), the protocol ensures that new changes incorporate and reference all of these heads. This process effectively merges several branches together, resulting in a unified and singular state for the object. During the retrieval of an object’s current state, the protocol begins with a specific head and follows Content IDs (CIDs) to construct the sequence of changes. If multiple heads or several possible ways to identify the previous change exist, a topological sort is employed. This sort relies on the hash values associated with the changes to determine their order, ultimately establishing the correct sequence of changes.33 This robust mechanism ensures eventual consistency and high fault tolerance within the system.

> 

> CRDTs enable a form of “asynchronous real-time” collaboration. Real-time collaboration traditionally implies a constant, synchronous connection to a central server for immediate updates and conflict resolution.31 Decentralized systems, however, face the challenge of ensuring consistency when devices are offline or experience network partitions.32 CRDTs allow replicas to converge to a consistent state eventually, regardless of the order of operations or temporary disconnections.32 They handle concurrent writes by merging multiple “heads”.33 This means Anytype can offer a real-time collaborative experience even when not continuously connected to a central server, and even when devices are temporarily offline.3 Changes made offline will seamlessly merge when connectivity is restored. Therefore, CRDTs provide the benefits of real-time co-editing (fluidity, responsiveness) with the robustness and resilience of a decentralized, offline-first system. This is a significant advancement over traditional models that typically break down without constant server connectivity. This capability allows for highly resilient and flexible collaborative workflows, particularly valuable for distributed teams, users in areas with unreliable internet, or scenarios requiring continuous productivity regardless of network state. It redefines what “real-time” means in a decentralized context.
