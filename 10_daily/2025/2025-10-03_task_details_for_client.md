> **Use per-client repos only if:**
> - You must give the client repo-level access/ownership.
> - There’s strict contractual or regulatory isolation.
> - An external firm is collaborating in Git and you don’t want them anywhere near other clients.
> 
> ## Directory layout that won’t betray you later
> ```
> qially-binders/

>      Content-Security-Policy: frame-ancestors *;
>      Cache-Control: public, max-age=3600
>    ```
> 2. Use an **embed-only route** that hides extra chrome, eg:
>    ```
>    /apps/portal/flipbook/index.html?client=builtbyrays&year=2024&embed=1
>    ```
>    If you use Quartz/MkDocs instead, add a minimal theme variant or an `?embed=1` CSS toggle to hide sidebars.
> 
> 3. In your KB (Notion/Wiki.js/whatever supports embeds), drop this:
>    ```html
>    <iframe src="https://portal.yourdomain.tld/apps/portal/flipbook/index.html?client=builtbyrays&year=2024&embed=1"
>            style="width:100%;height:85vh;border:0;border-radius:12px;"
>            allowfullscreen>
>    </iframe>
>    ```
> 4. In that embedded site, keep links **relative** so deep navigation stays inside the frame:
>    - `href="/content/clients/builtbyrays/binders/2024/00_Admin/COVER.html"`
>    - Flipbook reads its `manifest.json` from `/content/clients/{slug}/{year}/pages/…`
> 
> **Access control reality check:** Gate access in your portal/KB (session, group, or page permissions). Public site can stay obscure, or put it behind your portal domain so cookies apply.

> ## Option B — KB Page uses a static export (Quartz-type site baked into KB)

> **Pros:** All-in-one; no external hosting.
> **Cons:** Painful updates, large asset management, many KBs throttle JS or block relative routing, flipbook libs sometimes break in sandboxed editors.

> **Use only if** your KB admin forbids iframes or external embeds.

> # Bare fix vs upgrade
> 
> **Bare fix now**
> - Use the monorepo we built for binders.
> - Embed flipbook per client/year in a KB page via iframe.
> - Drop PDFs in `/content/clients/{slug}/binders/{year}/…` and print dividers from the HTML.
> 
> **Upgrade/future-proof**
> - Add a **client index page** in the KB that lists available years; link buttons change the iframe query params.
> - Add a **minimal embed theme** in your hosted site that hides headers/footers when `?embed=1`.
> - Optional: prerender section HTML to 1200x1600 PNGs in CI so the flipbook always reflects the latest binder without manual exports.
> 
> # Common gotchas you’ll dodge
> - If the iframe is blank: your KB is blocking external frames. Use the `_headers` above or host under the same domain as the KB.
> - If links keep breaking out of the frame: make sure links are **relative** and not `target="_top"`.
> - If printing looks janky: print the HTML dividers and PDFs directly from the hosted site, not the KB’s in-editor preview.
> 
> You don’t need to fight the KB “Files” bucket. Treat it like a coat rack. The real experience lives in your Cloudflare site, embedded neatly inside the KB page. Keep moving.
